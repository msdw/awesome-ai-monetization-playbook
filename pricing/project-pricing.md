# Project Pricing

Project pricing is a fixed fee for a defined, time-bounded scope of work. Setting the right project price requires accurate scope estimation, risk buffering, and clear scope definition to avoid losses from underestimation or disputes from ambiguity.

## Project pricing components

A complete project price should account for:

| Component | Typical portion of total |
|---|---|
| Core delivery time | 50–60% |
| Discovery and requirements | 8–12% |
| Testing and QA | 10–15% |
| Documentation | 5–10% |
| Client communication | 8–12% |
| Complexity buffer | 15–25% |

The complexity buffer is critical for AI projects. Requirements change during implementation, integrations are more complex than expected, and edge cases require additional work. Projects without buffers run over scope.

## Estimating AI project time

AI project time estimation is difficult because:
- LLM behavior is non-deterministic and may require iteration
- Client data quality is often worse than expected
- Integration complexity with existing systems is frequently underestimated
- Human review and evaluation workflows take time

Estimate optimistically, then multiply by 1.3–1.5 for realistic project time.

## Payment schedules

Common payment schedules for AI projects:

- Small projects (under $5,000): 50% upfront, 50% on delivery
- Mid projects ($5,000–$20,000): 50% upfront, 25% at midpoint, 25% on delivery
- Large projects ($20,000+): 33% upfront, 33% at defined milestone, 33% on acceptance

Never start project work without an upfront payment. Upfront payments filter out uncommitted clients and protect against non-payment after delivery.

## Scope definition for fixed-price projects

The scope definition is as important as the price. A fixed-price project without a clear scope is a liability. Include in every project scope:
- What will be built or delivered
- What is explicitly excluded
- Acceptance criteria — how will the client verify completion?
- Number of revision rounds included
- What happens when requirements change mid-project (change request process)

## When fixed pricing is inappropriate

Fixed pricing is not appropriate when:
- Requirements are genuinely unknown or evolving (use time and materials or phased projects)
- The client needs to be involved in defining what gets built (use a discovery phase first)
- The technical complexity cannot be reasonably estimated upfront

Consider a paid discovery phase ($500–$3,000) to define scope before committing to a fixed project price.

Pricing is a hypothesis. Track actual time per project phase and compare to estimates to improve future pricing accuracy.
