# Self-Serve Delivery

Self-serve delivery is a model where the buyer accesses, configures, and uses the product or service without requiring provider involvement. The provider builds the system and documentation; the buyer onboards and operates independently.

## How to implement

1. Build a product or tool that works without explanation — usability is the delivery model
2. Write comprehensive documentation covering common use cases and troubleshooting
3. Create in-product onboarding (tooltips, empty state guidance, setup wizards)
4. Build a help center or FAQ for common questions
5. Set up automated onboarding emails for new signups
6. Create video walkthroughs for complex features
7. Provide a support channel (email, chat) for edge cases that self-serve cannot resolve

## Documentation requirements

Self-serve products require:
- Quick start guide: get from zero to first value in 10 minutes
- Feature documentation: what each feature does and how to use it
- Integration guides: connecting to other tools
- Troubleshooting: common issues and solutions
- Change log: what changed in recent updates

Poor documentation is the most common reason self-serve products fail to activate users.

## Pros

- Scales without proportional provider time investment
- Users who onboard themselves have high ownership and often better retention
- Low cost per new user after initial product and documentation investment
- Enables global reach without timezone or language constraints

## Cons

- High upfront investment in product quality and documentation
- Support load can still be significant for complex products
- Not appropriate for high-complexity AI implementations requiring customization
- User activation requires excellent product design, not just documentation

## When to use

Self-serve delivery is appropriate for AI products with clear, contained use cases where the buyer can understand the value and configure the product independently. It is not appropriate for enterprise AI implementations, complex integrations, or regulated use cases requiring expert oversight.

## Metrics to track

- Time from signup to first meaningful action
- Self-serve activation rate (users who reach value without contacting support)
- Support ticket volume as a percentage of active users
- Documentation page views and search queries (reveals where users get stuck)
